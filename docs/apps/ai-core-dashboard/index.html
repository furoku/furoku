# 2026-05-16 レポート

## 今日何を研究したか
`verification_records` の `source`、`severity`、`passed` を planner scoring に戻し、未修正の可能性がある高重大度失敗を次回計画で優先する最小経路を研究した。

## 研究問い
AI Coreは `verification_records` の `source`、`severity`、`passed` をどう planner scoring に直接反映し、未修正の高重大度失敗を次回計画で優先できるか？

## 研究から立てた仮説
前回検証要約に `source` と `severity` を含め、`severity=high` かつ `passed=false` の文脈を planner が理由付きで加点すれば、検証証拠が単なるログではなく次行動の選択に影響する。ただし、今日の実装は構造化JSONの直接解析ではなくテキストトークン検出なので、長期的には誤加点を避ける構造化入力が必要。

## 計画
今日の能力増分は verification / planning / learning の接続改善。

成功条件:
- `research/2026-05-16.md` が research guard で valid。

## 分かったこと
- 前回検証要約に `source=...` と `severity=...` を含めることで、planner が検証の出所と重大度を最低限読めるようになった。
- `severity=high` と `passed=false` の文脈があると、verification 候補に `+4 high severity failed verification context` が残り、今日の run_cycle では `verification` が選択された。
- これにより、外部検証証拠を保存するだけだった前日から、検証証拠が planner の次行動を変える段階へ少し進んだ。

## 失敗・限界
- 今日の実装は `research_context` 文字列に含まれるトークンを読むだけで、同一検証レコード内の `passed=false` と `severity=high` の関係を厳密には保証しない。
- `resolved_at`、`failure_type`、未修正/修正済み状態は未実装。
- 高重大度失敗の扱いは固定加点で、失敗回数、古さ、影響範囲、再現性を考慮していない。
- Web研究は検索結果レベルに留まり、論文本文の詳細比較まではできていない。
- まだ初期足場であり、長期運用での自己改善・失敗回復・実用性は未確認。
